1 Requirements

1.1 Customer journey

1.1.1 Properties

Always identified by a slug, rather than by ID.

1.1.1.1 As a customer, I want to see the list of all my properties.

Route GET /properties.

1.1.1.2 As a customer, I want to see details of each of my properties.

Route GET /properties/:slug.

1.1.2 Receipts

Always identified by a combination of property-slug and date, rather than by ID.

1.1.2.1 As a customer, I want to be able to review all my income payments in one place.

Route GET /receipts.

1.1.2.2 As a customer, I want to be able to see all my income of a property in one place.

Route GET /property/:slug/receipts.

1.1.3 Incidents

Always identified by a combination of property-slug and date, rather than by ID.

1.1.3.1 As a customer, I want to be able to see all the incidents in one place.

Route GET /incidents.

1.1.3.2 As a customer, I want to be able to see all the incidents of a property in one place.

Route GET /property/:slug/incidents.

1.1.4 Account

1.1.4.1 As a customer, I want to be able to update my personal information

Bank connection, email etc.

Routes GET /account and PATCH /account.

Let's use PATCH rather than PUT, because we want to save each field once a field's focus is lost, rather than to have to use a save button.
